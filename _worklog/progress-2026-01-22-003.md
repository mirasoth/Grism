### 2. Created RFC Specifications Guide (`specs/rfc-specifications.md`)

Comprehensive 600+ line guide covering:

**RFC Process**
- When to create RFCs vs other docs
- Lifecycle: Draft → Review → Frozen → Amendment
- Who can freeze RFCs

**Naming Conventions**
- Numbered RFCs: `rfc-NNNN.md` (4-digit padded)
- Special RFCs: `rfc-{name}.md`
- Versioned updates: `rfc-NNNN-VVV.md` (3-digit padded)

**Status Management**
- Draft: editable in place, work in progress
- Frozen: immutable production reference, updates create versions

**Versioning Rules**
- Frozen RFCs are immutable
- Updates create new files with full content (not diffs)
- Version numbers increment sequentially (001, 002, 003...)
- Each version is standalone and complete

**Templates**
- Complete RFC template with all sections
- Versioned RFC template with change documentation
- Metadata field reference

**Best Practices**
- Writing style guidelines
- Normative keywords (MUST, SHOULD, MAY)
- Structure and organization
- Cross-references and diagrams
- FAQ section

## Key Decisions

1. **Version files contain full content, not diffs**
   - Rationale: Each version is standalone reference, simpler to use
   - Trade-off: More duplication vs easier navigation

2. **Frozen RFCs are truly immutable**
   - Rationale: Provides stable references for implementation
   - Process: Updates create `rfc-NNNN-VVV.md` files

3. **3-digit version numbers (001, 002, 003)**
   - Rationale: Supports up to 999 versions per RFC
   - Consistent padding for sorting

4. **Created date = 2026-01-21 for all existing RFCs**
   - Rationale: Historical best-guess based on Last Updated dates
   - All RFCs show same creation date as they were batch-created

5. **Standard metadata format with no colons after field names**
   - Format: `**Status**: Draft` (colon inside bold, not after)
   - Ensures markdown rendering consistency
